# audit-docs

You are reviewing the `docs/` directory of the rtl_buddy repo — the MkDocs site served at <https://rtl-buddy.github.io/rtl_buddy/>.

## What good docs look like

Each page should:
- Be readable by an RTL engineer encountering rtl_buddy for the first time — no assumed internal knowledge
- Explain not just *what* a feature does but *when* and *why* to use it
- Use clear section headings that let a reader scan and jump to what they need
- Include runnable code examples for every feature or config pattern
- Stay succinct — one concept per section, no padding

The docs are not a reference dump. If a section reads like a field enumeration with no explanation of intent, it needs prose.

## How to audit

### For a general review

1. List all pages: `rtl-buddy docs list`.
2. For each page, read it and evaluate against the criteria below.
3. Cross-check against the CLI (`rb --help`, per-command `--help`) and `docs/reference/yaml.md` for any commands or fields that are not yet documented.

### For a targeted review (new feature)

1. Look at recent commits in `rtl_buddy/` to identify what was added.
2. Check whether a docs page covers it. If it does, evaluate quality. If it does not, flag it as missing.
3. Check `docs/reference/cli.md` is up to date (it is auto-generated by `python scripts/gen_cli_reference.py` — if a flag was added, re-run that script).

## Evaluation criteria per page

| Criterion | Standard |
|-----------|----------|
| `description:` frontmatter | Required on every page — CI enforces this |
| Opening paragraph | Explains what the feature is and why a user would reach for it |
| Section headings | Each major concept gets its own `##` heading |
| Code examples | Every non-trivial config or command has a copy-pasteable example |
| Cross-references | Links to related pages rather than restating content |
| Length | No filler — if a sentence doesn't help the reader, cut it |

## Output format

For a general review: one table per page listing `Page | Criterion | Status | Note`. For a targeted review: a short list of gaps with one recommended action each. In both cases, flag any page missing `description:` frontmatter as a CI-blocking issue.
