# Testing Policy

This policy defines the evidence expected before merging a change to `rtl_buddy`.
The goal is repeatable confidence: every change should have the smallest test set that proves the changed behavior and protects the public CLI, YAML, logging, docs, and RTL workflow contracts.

## Success criteria

A change is ready when the author can state what behavior changed, which tests prove it, and which checks were intentionally skipped.
For code changes, that normally means the local unit test suite passes and at least one focused test covers the changed behavior.
For docs-only changes, that means docs frontmatter and docs rendering checks pass.
For workflow changes that affect simulator invocation, paths, artefacts, coverage, regressions, or agent-facing behavior, the change also needs downstream validation in `rtl-buddy-project-template`.

Do not treat "the command ran once" as enough evidence for reusable behavior.
Prefer a regression test that isolates the rule being changed, then use manual validation only for external simulator integration or end-to-end confidence.

## Required checks

Run the fastest complete local checks before opening or updating a PR:

```bash
uv run pytest
uv run python scripts/check_docs_frontmatter.py --check
uv run python scripts/gen_cli_reference.py --check
```

If a command, option, argument, callback, or help text changes, regenerate the CLI reference and commit it:

```bash
uv run python scripts/gen_cli_reference.py
```

If a check cannot run locally because a dependency or external tool is unavailable, record the exact command, the reason it could not run, and the substitute validation used.

## Test ownership

Every behavior change should add or update tests close to the behavior being changed.
Use the existing `tests/` suite for Python-level orchestration, config parsing, logging, docs access, artefact path handling, coverage path rewriting, setup failures, and skill installation behavior.

Add focused tests for:

- new CLI branches, flags, defaults, or error handling
- YAML schema or path-resolution behavior
- result classification, log output, or machine-readable event changes
- filelist, compile, simulation, coverage, regression, and post-processing path rules
- hooks such as sweep, preproc, and postproc
- docs access and bundled skill installation behavior

Do not add broad golden-output tests when a smaller assertion can prove the contract.
Mock or stub external tools when validating `rtl_buddy` orchestration logic; reserve real simulator runs for downstream validation.

## External tools

The unit suite should not require Verilator, VCS, Verible, LCOV, Coverview, or a licensed simulator to be installed.
Tests for wrappers around those tools should verify command construction, path handling, exit-code interpretation, and output parsing with temporary files or subprocess stubs.

When changing behavior that depends on a real external tool, run the smallest practical downstream command that exercises that tool and document the tool version when it matters.
Licensed or platform-specific checks may be PR notes rather than mandatory local checks, but the absence of that validation must be explicit.

## Downstream validation

Use `rtl-buddy-project-template` as the validation sandbox for behavior that spans a real RTL project layout.
For unreleased `rtl_buddy` changes, validate from the template repo's `dev/local-rtl-buddy` worktree and point its editable `rtl_buddy` source at the feature worktree under test.

Typical downstream checks include:

```bash
uv run rb filelist test_module -c design/example_block/src/models.yaml
uv run rb verible syntax design/example_block/src/test_module.sv
uv run rb regression -c regression.yaml
```

Run `test` and `randtest` from the suite directory unless the behavior being validated is specifically about explicit config paths:

```bash
cd design/example_block/verif
uv run rb test basic
uv run rb randtest basic --count 2
```

Select the smallest downstream command that proves the cross-repo behavior.
Full regressions are appropriate for changes to regression selection, per-test artefact layout, coverage merging, seed replay, or shared execution flow.

## Documentation

Update docs in the same PR when behavior, CLI help, YAML fields, artefact layout, validation workflow, or agent guidance changes.
User-facing behavior belongs under `docs/`; agent workflow guidance belongs in `src/rtl_buddy/skill/SKILL.md` only when agents need different operational instructions.

Every new docs page must include a non-empty `description:` frontmatter field.
Keep `docs/reference/cli.md` generated by `scripts/gen_cli_reference.py`; do not edit it by hand.

## Pull request notes

PR descriptions should include a short validation section with commands that were run and their outcomes.
If only a subset of checks was run, name the omitted checks and why they were not relevant or could not run.

Use this format:

```markdown
Validation:
- `uv run pytest`
- `uv run python scripts/check_docs_frontmatter.py --check`
- not run: downstream regression, docs-only change
```
